## Phase 2: 集成测试 (Medium)

### 2.1 Token统计精确性测试 ⭐⭐⭐
**目标**: 验证token统计与OpenClaw实际消耗一致

| 测试项 | 验收标准 | 优先级 |
|--------|----------|--------|
| 单次任务token | session_status返回值与实际一致 | P0 |
| 多轮对话token | 累计统计 = 各轮之和 | P0 |
| 异常任务处理 | timeout/failed时token=0或正确值 | P1 |
| 数据库记录 | BudgetTransaction与实际消耗一致 | P1 |
| 精度验证 | 0.0001 OC币精度无丢失 | P1 |

**验证方法**:
```
1. 记录OpenClaw账单系统的实际消耗
2. 对比OPC系统记录的token数
3. 误差应 < 1%（考虑四舍五入）
```

---

### 2.2 熔断系统全流程测试 ⭐⭐⭐
**目标**: 熔断触发→处理→恢复完整流程

#### 场景A: 100%熔断（预算耗尽）
```
前提: Agent月预算 = 100 OC币，已消耗 = 100 OC币
动作: 分配新任务（预算50 OC币）
预期: 
  - 任务状态 = paused
  - 熔断事件创建，状态 = triggered
  - Dashboard显示橙色警告条
  - Agent状态 = fused
```

#### 场景B: 熔断后加预算恢复
```
前提: 场景A后
动作: 执行"Add Budget"，增加预算200 OC币
预期:
  - Agent月预算 = 300 OC币
  - 任务自动恢复为pending
  - 熔断事件状态 = resolved
  - BudgetTransaction记录+200
```

#### 场景C: 熔断后任务拆分
```
前提: 场景A，任务预算50 OC币
动作: 拆分为2个子任务（20+25 OC币）
预期:
  - 原子任务状态 = split
  - 子任务创建，parent_task_id正确
  - 子任务可正常分配给该Agent（预算充足）
```

#### 场景D: 熔断后重新分配
```
前提: 场景A
动作: 重新分配给另一个Agent
预期:
  - 任务assigned_to更新
  - 原Agent释放任务
  - 新Agent检查预算通过
```

**边界测试**:
- 连续多次熔断（快速消耗预算）
- 0预算Agent触发熔断
- 超大预算（100万OC币）熔断计算

## Phase 3: 端到端流程测试 (Hard)

### 3.1 完整任务生命周期 ⭐⭐⭐⭐
**目标**: 从任务创建到完成的完整流程

```
流程图:
Partner创建任务 → 预检查预算 → 分配给员工 → 员工收到通知 → 
员工开始工作 → 实时Token统计 → 任务完成 → 生成日报 → 
预算更新 → 头像状态变化
```

**数据一致性验证点**:
1. 任务开始前: Agent.remaining_budget = X
2. 任务进行中: 实时扣减正确
3. 任务完成: Agent.remaining_budget = X - 实际消耗
4. BudgetTransaction记录完整
5. 日报数据 = 当日所有任务之和

---

### 3.2 Partner Agent自动创建员工 ⭐⭐⭐⭐
**目标**: 验证OpenClaw Agent创建流程

```
场景:
1. Partner在Dashboard点击"创建员工"
2. Partner调用OpenClaw API创建新Agent
3. 新Agent自动绑定到OPC系统
4. 新员工出现在员工列表
5. 新Agent可接收任务通知
```

**验证点**:
- OpenClaw中确实创建了Agent（通过API查询确认）
- Agent ID正确绑定到OPC Agent记录
- 新员工可正常通信
- 删除员工时可选是否删除OpenClaw Agent

---

### 3.3 头像与状态联动测试 ⭐⭐⭐⭐
**目标**: 头像状态动画与Agent状态同步

| Agent状态 | 头像效果 | 测试动作 |
|-----------|----------|----------|
| idle | 绿色慢呼吸 | 无任务时观察 |
| busy | 黄色中呼吸 | 分配任务后观察 |
| fused | 红色快脉冲 | 触发熔断后观察 |
| offline | 灰色静态 | 模拟Agent离线 |

**多状态切换测试**:
```
idle → busy: 分配任务，黄色呼吸在3秒内生效
busy → fused: 预算耗尽，红色脉冲立即生效
fused → idle: 加预算恢复，绿色呼吸生效
```

## Phase 4: 压力与异常测试 (Very Hard)

### 4.1 并发压力测试 ⭐⭐⭐⭐⭐
**目标**: 系统在高并发下的稳定性

| 测试项 | 场景 | 验收标准 |
|--------|------|----------|
| 并发任务分配 | 10个任务同时分配给同一Agent | 无数据竞争，预算计算正确 |
| 并发消息发送 | 100条消息同时发给Partner | 无丢失，顺序可能乱但不丢 |
| 头像上传并发 | 10个头像同时上传 | 文件名冲突正确处理 |
| 数据库连接池 | 50并发查询 | 无连接泄漏，响应<1s |
| 图表数据量 | 365天历史数据 | 渲染不卡顿，数据正确 |

---

### 4.2 异常恢复测试 ⭐⭐⭐⭐⭐
**目标**: 系统在异常情况下的自愈能力

| 异常场景 | 模拟方法 | 预期行为 |
|----------|----------|----------|
| 数据库断开 | 停止PostgreSQL | 优雅降级到SQLite或报错提示 |
| OpenClaw不可达 | 断网 | 任务标记failed，不重试死循环 |
| 磁盘满 | 填充/tmp | 上传头像优雅失败，提示用户 |
| 内存不足 | 限制容器内存 | 大图表渲染失败但不崩溃 |
| 时钟回拨 | 修改系统时间 | 时间戳不混乱，日志正确 |

---

### 4.3 数据一致性压力测试 ⭐⭐⭐⭐⭐
**目标**: 长时间运行后数据仍然一致

**测试方案**:
```
1. 创建5个Agent，各1000 OC币预算
2. 自动化运行100个任务（随机预算10-100）
3. 随机触发熔断和各种处理动作
4. 运行24小时后验证:
   - 所有Agent预算总和 = 初始总和 - 所有任务消耗
   - BudgetTransaction记录数 = 实际任务数
   - 无负预算Agent
   - 无孤儿任务（assigned_to不存在）
```
