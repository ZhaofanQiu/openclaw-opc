# OpenClaw OPC 开发手册

**版本**: v1.0  
**生效日期**: 2026-03-23  
**适用范围**: 所有 OpenClaw OPC 项目开发者

---

## 核心理念

### 一句话原则

> **先跑通，再完善；先核心，再周边；真实可用 > 表面完整。**

### 开发哲学

1. **虚假繁荣是项目最大的敌人**
   - UI 好看但功能不可用 = 垃圾
   - API 齐全但流程不通 = 垃圾
   - 文档完美但代码不实 = 垃圾

2. **功能必须真实可用**
   - 每个功能提交前必须能跑通端到端流程
   - 禁止只写"壳"不写"核"
   - 禁止用假数据串流程

3. **阶段性交付**
   - 一层跑通，再下一层
   - 一个功能真实可用，再开发下一个
   - 宁可功能少，不可功能假

## 一、文档规范

### 1.1 文档结构（强制）

```
docs/
├── README.md              # 项目入口（必须简洁明了）
├── DESIGN.md              # 产品设计（权威设计文档）
├── ARCHITECTURE.md        # 架构设计（技术架构）
├── ROADMAP.md             # 路线图（里程碑规划）
├── API.md                 # API 文档（自动生成+手动补充）
├── DEPLOYMENT.md          # 部署指南
├── CONTRIBUTING.md        # 本开发手册
├── CHANGELOG.md           # 变更日志
└── archive/               # 归档目录
    ├── 2026-03/           # 按月份归档
    └── obsolete/          # 过时文档
```

**禁止**: 在根目录创建临时文档、测试报告、bug 分析等。

### 1.2 文档编写原则

| 原则 | 说明 |
|------|------|
| **单一权威** | 同一主题只能有一个权威文档 |
| **及时归档** | 临时文档完成后立即归档 |
| **拒绝重复** | 不重复写已有文档的内容 |
| **代码即文档** | API 优先用代码注释+自动生成 |

### 1.3 文档生命周期

```
创建 → 评审 → 定稿 → 维护 → 归档
              ↑
         过时/被替代时
```

**归档标准**:
- 版本迭代后，旧版本设计文档
- 临时分析文档（bug 分析、测试报告）
- 被新文档替代的旧文档

## 二、代码规范

### 2.1 架构原则

#### 三层架构（强制遵守）

```
┌─────────────────────────────────────────┐
│  Layer 1: 核心层                         │
│  - 必须最先跑通                          │
│  - 所有其他层依赖它                      │
├─────────────────────────────────────────┤
│  Layer 2: 管理层                         │
│  - 依赖 Layer 1                          │
│  - 业务逻辑层                            │
├─────────────────────────────────────────┤
│  Layer 3: 体验层                         │
│  - 依赖 Layer 1+2                        │
│  - 游戏化、氛围                          │
└─────────────────────────────────────────┘
```

**铁律**: Layer N 不跑通，不开发 Layer N+1。

### 2.2 目录结构

```
backend/src/
├── routers/               # 路由层
│   ├── agents.py          # 员工（含技能、成长）
│   ├── tasks.py           # 任务（含步骤、依赖）
│   ├── workflows.py       # 工作流
│   ├── messages.py        # 消息（含通知）
│   ├── manuals.py         # 手册
│   ├── budgets.py         # 预算
│   └── ...                # 其他（不超过10个）
├── services/              # 服务层
│   ├── agent_service.py
│   ├── task_service.py
│   ├── workflow_service.py
│   └── ...                # （不超过15个）
├── models/                # 数据模型
├── core/                  # 核心逻辑
│   ├── agent_interaction.py   # Agent 交互（最重要）
│   ├── skill_framework.py     # Skill 框架
│   └── manual_application.py  # 手册应用
├── utils/                 # 工具函数
└── tests/                 # 测试（必须存在）
```

**红线**:
- Router 文件不超过 10 个
- Service 文件不超过 15 个
- 新文件创建前必须评审

### 2.3 编码规范

#### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件 | 小写下划线 | `task_service.py` |
| 类 | 大驼峰 | `TaskService` |
| 函数 | 小写下划线 | `assign_task()` |
| 常量 | 大写下划线 | `MAX_RETRY_COUNT` |
| API 路径 | 小写中划线 | `/api/task-steps` |

#### 函数设计

```python

    Returns:
        更新后的任务对象
        
    Raises:
        ValueError: 参数错误
        TaskNotFoundError: 任务不存在
        AgentNotAvailableError: 员工不可用
    """
    pass

# ❌ 坏的函数：职责混乱，难以测试
def handle_task_operation(operation: str, data: dict):
    # 分配、完成、返工混在一起
    pass
```

#### 错误处理

```python
# ✅ 统一错误处理
from fastapi import HTTPException

class OPCException(Exception):
    """业务异常基类"""
    def __init__(self, code: str, message: str, status_code: int = 400):
        self.code = code
        self.message = message
        self.status_code = status_code

# Service 层抛出业务异常
```

## 三、开发流程

### 3.1 需求阶段

#### 需求评审清单

- [ ] 这个功能属于哪一层？（Layer 1/2/3）
- [ ] 依赖的功能是否已跑通？
- [ ] 是否有重复功能？
- [ ] 是否可以用现有功能组合实现？

**禁止**: 跨层开发、依赖未跑通功能、重复造轮子。

### 3.2 设计阶段

#### 设计文档要求

1. **必须写设计文档的场景**:
   - 新功能涉及多个模块
   - 修改核心架构
   - 新增数据模型

2. **设计文档内容**:
   ```markdown
   # 功能设计：[功能名称]
   
   ## 背景
   为什么需要这个功能？
   
   ## 目标
   具体要实现什么？
   
   ## 方案
   技术方案概述
   
   ## 数据模型
   新增/修改的模型
   
   ## API 设计
   接口定义
   
   ## 验收标准
   如何验证功能正确？
   ```

### 3.3 开发阶段

#### 开发 checklist

- [ ] 已阅读相关设计文档
- [ ] 依赖功能已跑通
- [ ] 已编写单元测试（核心逻辑）
- [ ] 代码符合规范
- [ ] 已自测通过

### 3.4 测试阶段

#### 测试要求（强制）

| 层级 | 要求 | 说明 |
|------|------|------|
| 单元测试 | 核心逻辑必须覆盖 | 特别是 Layer 1 |
| 集成测试 | 关键流程必须覆盖 | 端到端测试 |
| 人工测试 | 所有功能必须测试 | 不能只看代码 |

#### 端到端测试定义

**什么算"跑通"**:

```
❌ 假的跑通:
- API 返回 200
- 数据库有记录
- 页面能显示

✅ 真的跑通:
- 用户操作 → 系统处理 → Agent 执行 → 返回结果
- 整个链条真实可用
- 不是假数据、不是 mock
```

**示例**:

```
任务分配功能的真的跑通:
1. 用户创建任务
2. 分配给 Agent
3. Agent 真正被唤醒
4. Agent 真正执行任务
5. Agent 返回真实结果
6. 任务状态正确更新
7. 预算正确计算

如果第 3-4 步是假的，就不算跑通。
```

### 3.5 提交阶段

#### Commit 规范

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type**:
- `feat`: 新功能
- `fix`: 修复
- `refactor`: 重构
- `docs`: 文档
- `test`: 测试
- `chore`: 杂项

**示例**:
```
feat(task): 实现任务分配功能

- 支持单员工任务分配
- 自动创建任务步骤
- 集成手册生成

Closes #123
```

#### PR 要求

1. **PR 描述必须包含**:
   - 功能说明
   - 变更清单
   - 测试方法
   - 截图（UI 变更）

2. **Review  checklist**:
   - [ ] 代码规范检查
   - [ ] 功能真实可用验证
   - [ ] 文档同步更新
   - [ ] 无重复功能

## 四、防虚假繁荣机制

### 4.1 什么是虚假繁荣

| 类型 | 表现 | 危害 |
|------|------|------|
| API 空壳 | 接口返回 200，但无实际逻辑 | 以为功能完成，实际不可用 |
| UI 空壳 | 页面好看但按钮无功能 | 用户体验割裂 |
| 流程假通 | 用假数据串流程 | 无法发现真实问题 |
| 文档过度 | 写了很多文档但代码不实 | 精力分散，本末倒置 |

### 4.2 识别虚假繁荣

**自我检查**:

```
问自己的问题:
1. 这个功能用户现在能真实使用吗？
2. 如果我现在演示，是真实流程还是假数据？
3. 如果我删掉 mock，功能还能跑通吗？
4. 我的测试是测真实逻辑还是测返回值？
```

**团队检查**:

```
Code Review 时必须验证:
1. 是否调用了真实的 Agent API？
2. 是否处理了真实的响应？
3. 是否有端到端的测试证据？
```

### 4.3 禁止清单

❌ **绝对禁止**:

- 用假数据串流程
- 只写 API 不写实现
- 只写 UI 不写后端
- 写文档但不更新代码
- 声称功能完成但没有端到端测试

✅ **必须做到**:

- 每个功能都要有真实可用的证据
- 代码提交前自己先跑通端到端
- 复杂功能先写设计文档
- 定期删除无用代码

## 六、协作规范

### 6.1 分支策略

```
main                    # 主分支，稳定版本
├── develop             # 开发分支
├── feature/xxx         # 功能分支
├── refactor/xxx        # 重构分支
└── hotfix/xxx          # 紧急修复
```

### 6.2 协作流程

1. 从 `develop` 创建功能分支
2. 开发完成后提交 PR 到 `develop`
3. Code Review 通过
4. 合并到 `develop`
5. 定期合并 `develop` 到 `main`

### 6.3 沟通规范

- **技术讨论**: 在 PR 中进行
- **设计讨论**: 先写文档，再讨论
- **问题反馈**: 带复现步骤和日志

## 七、违规处理

### 7.1 常见违规

| 违规 | 处理 |
|------|------|
| 虚假功能（API/UI 空壳） | PR 拒绝，要求补全 |
| 跨层开发 | 要求拆分，先跑通依赖 |
| 重复功能 | 要求合并或删除 |
| 文档混乱 | 要求整理后再提交 |
| 无测试 | 要求补充测试 |

### 7.2 严重违规

以下情况可能导致代码回滚：
- 虚假功能合并到主分支
- 未评审直接合并
- 明知依赖未跑通仍强行开发
