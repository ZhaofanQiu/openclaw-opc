# v0.4.2 实际应用场景测试方案

## 概述

本测试方案通过4个真实业务场景，端到端验证 v0.4.2 (P0-P2) 的工作流功能。

---

## 场景1: 内容创作流水线 📝

**业务背景**: 从选题到发布的完整内容创作流程

### 参与Agent
| 角色 | Agent | 职责 |
|------|-------|------|
| 策划 | ContentPlanner | 选题策划、大纲设计 |
| 写手 | Writer | 文章撰写 |
| 编辑 | Editor | 审核、修改建议 |
| 发布 | Publisher | 格式转换、多渠道发布 |

### 工作流步骤
```
Step 1: 策划 → 输出选题和大纲 (预算: 0.5)
Step 2: 写手 → 撰写初稿 (预算: 2.0)
Step 3: 编辑 → 审核反馈 (预算: 0.8)
  └─ 返工路径 → 回到Step 2 (最多2次)
Step 4: 发布 → 多平台发布 (预算: 0.3)
```

### 测试的功能点
| 功能 | 验证点 |
|------|--------|
| P0 基础工作流 | 4步骤串行执行 |
| P0 返工机制 | Step 3 → Step 2 返工 |
| P0 预算追踪 | 每步Token消耗记录 |
| P0 执行日志 | 完整执行历史 |
| P1 条件分支 | 审核通过/返工判断 |
| P2 时间线 | 查看每步耗时和返工记录 |
| P2 分析 | 返工率统计、平均创作周期 |

    "word_count": 2000,
    "style": "技术博客",

  }
}

## 场景2: 代码审查流水线 🔍

**业务背景**: 代码提交后的自动化审查流程

### 参与Agent
| 角色 | Agent | 职责 |
|------|-------|------|
| 静态检查 | Linter | 代码规范检查 |
| 安全扫描 | SecurityBot | 漏洞扫描 |
| 架构审查 | Architect | 设计模式审查 |
| 测试生成 | TestWriter | 生成单元测试 |

### 工作流步骤
```
Step 1: Linter → 规范检查 (预算: 0.3)
  └─ 失败 → 直接结束 (失败状态)
Step 2: SecurityBot → 安全扫描 (预算: 0.5)
  └─ 高风险 → 结束并告警
Step 3: Architect → 架构审查 (预算: 0.8)
  └─ 返工 → 回到开发者 (标记返工)
Step 4: TestWriter → 生成测试 (预算: 0.5)
```

### 测试的功能点
| 功能 | 验证点 |
|------|--------|
| P0 失败处理 | Step 1/2 失败直接结束 |
| P0 状态流转 | pending → assigned → in_progress → completed/failed |
| P1 并行检查 | Linter + SecurityBot 并行（如支持）|
| P1 条件执行 | 根据风险等级决定后续步骤 |
| P2 模板 | 保存为 "Code Review Template" |
| P2 评分 | 审查完成后对模板评分 |

### 测试数据
```json

    "repo": "github.com/company/project",
    "pr_id": "#123",
    "files": ["src/auth.py", "src/api.py"],
    "language": "python"
  }
}
```

## 场景3: 客户服务响应 🤖

**业务背景**: 客户咨询的自动分类和路由处理

### 参与Agent
| 角色 | Agent | 职责 |
|------|-------|------|
| 分类器 | Classifier | 问题分类（技术/商务/投诉）|
| 技术支持 | TechSupport | 回答技术问题 |
| 销售 | Sales | 处理商务咨询 |
| 客诉 | ComplaintHandler | 处理投诉 |

### 工作流步骤
```
Step 1: Classifier → 分类问题 (预算: 0.2)
Step 2: 路由 → 根据类型分配
  ├─ 技术 → TechSupport (预算: 1.0)
  ├─ 商务 → Sales (预算: 0.8)
  └─ 投诉 → ComplaintHandler (预算: 1.2)
Step 3: 质检 → QA审核回复 (预算: 0.3)
  └─ 返工 → 回到对应Agent
```

### 测试的功能点
| 功能 | 验证点 |
|------|--------|
| P1 条件分支 | 根据分类结果路由到不同Agent |
| P1 动态步骤 | 根据类型选择执行路径 |
| P2 模板Fork | 基于基础模板Fork各类型专用模板 |
| P2 使用统计 | 各模板使用次数统计 |
| P2 员工排名 | 各Agent响应效率对比 |

## 场景4: 数据报告生成 📊

**业务背景**: 从原始数据到可视化报告的自动化流程

### 参与Agent
| 角色 | Agent | 职责 |
|------|-------|------|
| 数据清洗 | DataCleaner | 清洗原始数据 |
| 分析师 | Analyst | 数据分析和洞察 |
| 设计师 | Designer | 图表设计 |
| 报告员 | Reporter | 撰写报告结论 |

### 工作流步骤
```
Step 1: DataCleaner → 数据清洗 (预算: 0.5)
Step 2: Analyst → 数据分析 (预算: 1.5)
  ├─ 子任务1: 趋势分析
  ├─ 子任务2: 异常检测
  └─ 子任务3: 预测建模
Step 3: Designer → 可视化设计 (预算: 0.8)
Step 4: Reporter → 报告撰写 (预算: 1.0)
  └─ 返工 → 补充分析 (回到Step 2)
```

### 测试的功能点
| 功能 | 验证点 |
|------|--------|
| P0 多步骤执行 | 4步骤完整执行 |
| P0 预算熔断 | 设置总预算上限，触发熔断 |
| P1 子工作流 | Step 2 拆分为3个子分析任务 |
| P2 模板市场 | 发布为 "数据报告模板" |
| P2 评分评论 | 用户评分和反馈 |
| P2 分析看板 | 平均报告生成时间、返工率 |
