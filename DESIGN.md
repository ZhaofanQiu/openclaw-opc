# OpenClaw OPC 设计文档

> 实用性 × 游戏性：让 AI Agent 成为你的虚拟团队

---

## 核心理念

### 一人公司操作系统

将 OpenClaw Agent 组织成虚拟公司，让 AI 成为可管理、可协作、可成长的"员工"。

### 双轮驱动设计

```
┌─────────────────────────────────────────┐
│           OpenClaw OPC                  │
├─────────────────┬───────────────────────┤
│     实用性       │        游戏性         │
├─────────────────┼───────────────────────┤
│ • 项目推进      │ • 像素办公室          │
│ • 任务分配      │ • 员工养成            │
│ • 多 Agent 协同 │ • 氛围系统            │
│ • 成本控制      │ • 可视化反馈          │
└─────────────────┴───────────────────────┘
```

**设计原则**:
1. **实用优先** - 游戏化是调味剂，核心是高效完成项目
2. **安全第一** - 预算熔断、超时保护，防止 Token 无限消耗
3. **透明度** - Token 消耗实时可见，成本可控
4. **渐进游戏化** - 不强制，让用户自然沉浸

## 实用性系统

### 1. 项目管理

**项目生命周期**:
```
立项 → 规划 → 执行 → 监控 → 验收 → 归档
```

**核心功能**:
- **里程碑管理**: 时间线、依赖关系
- **任务池**: 待分配、进行中、已完成
- **进度追踪**: 实时看板、燃尽图
- **预算控制**: 项目级别预算上限

### 2. 任务系统

**任务状态流**:
```
待分配 → 已分配 → 进行中 → 待验收 → 已完成
   ↓         ↓         ↓         ↓
 可取消   可转派    可暂停    可返工
```

**任务类型**:

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| 标准任务 | 常规流程 | 日常开发、文档 |
| 研究任务 | 结果不确定 | 技术调研、探索 |
| 紧急任务 | 限时高优 | Bug 修复、上线 |

**任务分配策略**:
1. 用户创建任务，设定预算和期望
2. 系统推荐合适的 Agent（基于技能和负载）
3. 用户确认或调整分配
4. Agent 执行并主动汇报进度

### 3. 多 Agent 协同

**协作模式**:

| 模式 | 说明 | 示例 |
|------|------|------|
| 串行 | A 完成 → B 开始 | 设计 → 开发 → 测试 |
| 并行 | A 和 B 同时进行 | 前端和后端并行开发 |
| 评审 | A 产出 → B 评审 | Code Review |

**协作协议**:
```
1. Agent A 完成任务步骤
2. A 通过 opc_report_task_result() 汇报
3. OPC 通知 Agent B（通过 sessions_send）
4. B 通过 opc_get_current_task() 获取任务
5. B 读取手册，开始执行
```

### 4. 预算控制系统

**三级熔断**:

| 级别 | 触发条件 | 动作 |
|------|---------|------|
| 警告 | 消耗达到 80% | UI 提醒，员工显示"注意" |
| 暂停 | 消耗达到 100% | 任务暂停，等待决策 |
| 熔断 | 消耗达到 150% | 强制停止，任务失败 |

**熔断后选项**:
- 追加预算（从公司储备调拨）
- 拆分任务（把大任务拆小）
- 换人重做（换一个 Agent 尝试）
- 取消任务（止损）

## 游戏性系统

### 1. 像素办公室

**设计理念**:
- 有温度的可视化，区别于冷冰冰的表格
- 一眼看出团队状态和氛围
- 8-bit 像素风格，致敬经典

**核心界面**:

```
┌──────────────────────────────────────┐
│  💼 OpenClaw OPC 办公室               │
├──────────────────────────────────────┤
│                                      │
│  ┌────┐  ┌────┐  ┌────┐  ┌────┐     │
│  │ 🧑‍💻│  │ 👩‍💻│  │ 🧑‍💼│  │ 👨‍💻│     │
│  │小明 │  │小红 │  │Partner│  │小李 │     │
│  │😊  │  │😐  │  │😊  │  │😔  │     │
│  └────┘  └────┘  └────┘  └────┘     │
│                                      │
│  预算剩余: ¥12,500 (62%)             │
│  进行中: 3 任务 | 待验收: 2 任务      │
│                                      │
└──────────────────────────────────────┘
```

**员工状态可视化**:
- **空闲**: 正常待机动画
- **工作中**: 打字动画
- **疲惫**: 偶尔打哈欠
- **预算紧张**: 显示警告图标

### 2. 员工系统

**员工属性**:
```
员工 = {
  基本信息: {姓名, 头像, emoji, 职位等级},
  预算: {月薪, 已消耗, 剩余},
  心情: 基于预算的视觉反馈,
  技能: {技能名: 熟练度},
  手册: [L2技能手册],
}
```

**职位体系**:
```
实习生 (Lv.1) → 专员 (Lv.2) → 资深 (Lv.3) → 专家 (Lv.4) → 合伙人 (Lv.5)
```

**特殊角色：合伙人 (Partner)**
- 用户的第一个 OpenClaw Agent 自动成为合伙人
- 协助 CEO 管理公司、处理异常
- 可通过悬浮窗快速沟通
- 拥有最高权限，但受安全规则约束

### 3. 心情系统 = 预算仪表盘

**核心设计**:
```
心情值 = 剩余预算比例
```

| 心情 | 预算状态 | 视觉反馈 |
|------|---------|---------|
| 😊 很好 | > 60% | 正常动画 |
| 😐 一般 | 30-60% | 偶尔提醒 |
| 😔 疲惫 | < 30% | 显示"需要休息" |

**关键点**: 心情只影响氛围，不影响实际产出质量。

### 4. 手册系统（员工养成）

**三层手册架构**:

| 层级 | 类型 | 说明 | 效果 |
|------|------|------|------|
| L1 | 岗位通用手册 | 每个岗位 1 本 | 基础工作规范 |
| L2 | 专项技能手册 | 员工最多 2 本 | 决定"能做什么"和效率 |
| L3 | 项目专属手册 | 动态生成 | 项目特有规范 |

**手册等级与效果**:

| 等级 | 熟练度 | 返工率 | 特殊效果 |
|------|--------|--------|---------|
| 入门级 | 0-30 | 25%+ | 基础可用 |
| 初级 | 31-60 | 20% | 自动生成 FAQ |
| 中级 | 61-80 | 15% | 可指导实习生 |
| 高级 | 81-95 | 10% | 可参与评选 |
| 专家级 | 96-100 | 5% | 可发布社区 |

**手册健康度**:
- 根据实际任务完成情况动态更新返工率
- 高等级但高返工率的手册会被警告

### 5. 事件系统（氛围营造）

**触发条件** (仅 IDLE 状态):
- 心情 < 30%
- 工作时长 > 4 小时
- 任务失败/成功
- 协作冲突

**频率控制**:
- 每个员工每天最多 2 个事件
- 冷却期 4 小时
- 用户可设置"勿扰模式"

**事件示例**:
- "小明完成了一个困难任务，心情变好"
- "小红需要更多预算来完成任务"
- "合伙人建议调整项目优先级"

## Skill 驱动架构

### Agent 与 OPC 的交互

```
┌──────────┐     sessions_send      ┌──────────┐
│   OPC    │ ─────────────────────→ │  Agent   │
│          │   (任务分配消息)        │          │
│          │                        │          │
│          │ ←───────────────────── │          │
│          │   opc_* API calls      │          │
│          │   - get_current_task() │          │
│          │   - read_manual()      │          │
│          │   - report_result()    │          │
└──────────┘                        └──────────┘
```

### Skill API（Agent 可调用的方法）

| 方法 | 用途 |
|------|------|
| `opc_get_current_task()` | 获取当前任务详情 |
| `opc_read_manual()` | 读取工作手册 |
| `opc_report_task_result()` | 汇报任务结果 |
| `opc_get_budget()` | 查询预算状态 |

## 核心界面

### 1. Dashboard（仪表盘）

**功能**:
- 数据概览：员工数、任务数、预算

## 安全与限制

### Agent 行为边界

**允许**:
- 读取工作目录
- 写入工作目录
- 调用 OPC Skill API
- 向特定同事发送消息（限 3 轮）

**禁止**:
- 修改系统配置
- 访问其他员工私有目录
- 创建新 Agent（仅合伙人可）
- 删除历史记录

### 时间保护

- 连续工作 > 2 小时 → 提醒休息
- 连续工作 > 4 小时 → 强制暂停
- 单次对话 > 500 Token → 截断

## 差异化亮点

| 特点 | 说明 |
|------|------|
| **模块化架构** | 四包独立，易于维护和扩展 |
| **真实成本意识** | Token 消耗转化为虚拟币，预算硬边界 |
| **Skill 驱动** | Agent 主动获取任务，确保执行闭环 |
| **像素办公室** | 有温度的可视化，区别于冷冰冰的表格 |
| **游戏化但不干扰** | 氛围系统不影响实际产出 |
| **合伙人机制** | Agent 辅助管理，降低用户负担 |
